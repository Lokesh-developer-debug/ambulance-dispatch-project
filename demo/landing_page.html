Core Features

Everything an AI agent needs
to learn dispatch optimization

Built on FastAPI with full OpenEnv compatibility. Every feature is designed to make the dispatch problem realistic and challenging.

🚑
Smart Auto Dispatch
Priority-based assignment (CRITICAL → MEDIUM → LOW) with nearest ambulance selection. All ambulances dispatched simultaneously in a single step.
👥
Multi-Pickup Support
Single ambulance can pick up two nearby patients in one trip. Checks distance radius (15 units) and remaining survival time before allowing pickup.
📞
Dynamic Spawning
New patients appear during the episode like real 108 calls. Spawn probability increases across early, middle, and late phases.
💥
Cluster Spawning
10% chance of accident scene with 2–3 patients spawning at the same location. Perfect for multi-pickup scenarios simulating mass casualty events.
👁️
Partial Observability
Patients only become visible when an ambulance is within the visibility radius. Once seen, always tracked. Configurable radius per episode.
🏥
Hospital Management
Hospitals track available regular beds and ICU capacity. Critical patients need ICU beds. Nearest hospital with capacity is auto-selected.
⚡
Simultaneous Actions
step_multiple() applies all dispatch actions at once before the world advances. Prevents critical patients dying while waiting for sequential dispatch.
🎯
Configurable Difficulty
3 preset task levels: Easy (3 amb, 4 patients), Medium (3 amb, 6 patients), Hard (2 amb, 8 patients). All parameters fully customizable per request.
📊
Rich Episode Logging
Full episode log with per-step actions, rewards, phase info, spawn events, multi-pickup flags, and final statistics including delivery rate.
Dynamic Spawning

Three phases of escalating
emergency pressure

Patient arrival patterns change throughout the episode, mimicking real 108 call center patterns with progressive severity escalation.

Steps 0 – 20
Early Phase
Spawn probability 20%
CRITICAL rate 10%
MEDIUM rate 40%
LOW rate 50%
Calm period. Mostly minor injuries. Ambulances can establish initial positions and handle the starting patients.
Steps 21 – 40
Middle Phase
Spawn probability 50%
CRITICAL rate 30%
MEDIUM rate 50%
LOW rate 20%
Calls intensify. More critical cases. Agent must balance ongoing deliveries with newly arriving emergencies requiring triage decisions.
Steps 41 – 60
Late Phase
Spawn probability 70%
CRITICAL rate 50%
MEDIUM rate 40%
LOW rate 10%
Peak emergency. Mostly critical patients with tight 5-step death windows. Maximum pressure tests the agent's prioritization under resource constraints.
REST API

8 endpoints powering
the full RL loop

Standard OpenEnv-compatible HTTP API. Supports single-step control, multi-action batching, and full automatic episode execution.

POST /reset
Start a new episode. Spawns ambulances near hospitals, patients at random positions, and initializes all tracking state.
dynamic-spawning partial-observability configurable
POST /step
Apply a single action (DISPATCH / ROUTE_TO_HOSPITAL / WAIT). Returns updated observation, reward, done flag, and info dict.
movement multi-pickup death-timer reward
POST /auto_dispatch
Run a complete episode automatically. Smart dispatcher handles all decisions — priority routing, multi-pickup, hospital selection, and reassignment.
smart-dispatch multi-pickup cluster-spawn statistics
GET /observation
Retrieve the current observation without advancing the environment. Returns all visible patients, ambulance positions, hospital capacity, and phase.
visibility-filter phase-info spawn-count
GET /state
Returns episode metadata: unique episode ID, current step count, active phase, and total patients spawned so far.
episode-id step-count phase
GET /action_space
Returns all valid action types, ambulance IDs, patient IDs (including dynamic spawns up to max), and hospital IDs for the current configuration.
action-types valid-ids dynamic-ids
GET /observation_space
Returns space metadata: map dimensions, max ambulances, max patients, hospital count, and whether dynamic spawning is enabled.
map-size max-entities spawn-flag
GET /health
Liveness probe returning 200 OK. Used by Docker health checks and Hugging Face Spaces deployment monitoring.
docker huggingface monitoring
